# Fix: SplitMode drag-replace works only once, then reverts to adding

## Symptom
In SplitMode, dragging a file onto an existing card zone:
- 1st time: correctly replaces the target card (card count stays same)
- 2nd+ times: ADDS a new card instead of replacing (card count increases)

## Root cause hints
The `isDragging` ref in FileTree.tsx is set to true in `onDragStart` and reset in `onDragEnd`. However, after the first successful drag, something prevents `onDragEnd` from firing or the ref isn't resetting properly, causing subsequent drags to skip the onClick guard but also potentially passing wrong state.

## Files to investigate
- `frontend/src/features/viewer/FileTree.tsx` — the `isDragging` ref, onDragStart, onDragEnd, onClick handlers
- `frontend/src/features/viewer/ViewerPage.tsx` — SplitGrid drop handler, handleFocusFile, splitMode state, setNodes logic

## What to check
1. Does `onDragEnd` actually fire after drop completes? Add `console.log` to verify
2. After first replacement, does SplitGrid's `cardCount` update correctly? (should reflect new node count)
3. Does `splitMode` remain true after setNodes runs?
4. Is the SplitGrid component re-mounted (losing drag state) after the first replacement?
5. The `nodes` array — after replacement, does it have the correct count? Check `nodes.map(n => n.id)` length

## Expected behavior
- Every drag-drop onto an occupied zone should call `handleFocusFile(fileId, replaceIdx)` with `replaceIdx` = the zone index
- This should replace the card at that position, keeping total card count unchanged
- Clicking a file should ADD a new card (up to max 4)
