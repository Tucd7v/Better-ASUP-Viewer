Fix file-drop-to-replace in Grid mode after card swap.

## Current bug
After swapping two cards via grip drag, dragging a new file from FileTree to replace a card at visual position X replaces the wrong card.

## Root cause
`handleDrop` maps visual position → parent gridNodes index → passes to `onDropFile`. `onDropFile` replaces in the parent nodes array. But after a swap, the parent order != visual order (visual order is from `cardOrder`). The new card appears at the parent index, not the visual position.

## Fix
Instead of translating to parent index, SplitGrid should directly replace the card in its own `cardOrder` at the visual slot. Don't call `onDropFile`.

### In SplitGrid
`handleDrop` already computes which card is at the visual slot. Instead of calling `onDropFile`:
1. Read the fileId from dataTransfer
2. Tell the parent to spawn a new card node (call a `spawnCard` callback that adds the node to parent array)
3. When the parent adds the node, SplitGrid detects the new node in `gridNodes` via `currentGridNodeIds` change
4. Then update `cardOrder` to put the new card's ID at the visual slot

Simpler approach: Add a callback `onSpawnAndReplace: (fileId: string, replaceNodeId: string) => void` that:
- Parent spawns a new node and adds it
- Parent replaces the old node (by ID) with the new node
- SplitGrid's syncCardOrder handles the rest

OR simplest: pass `onSpawnCard: (fileId: string) => string` that spawns and returns the new nodeId. Then SplitGrid does:
1. const newNodeId = onSpawnCard(fileId) 
2. setCardOrder(prev => { const next = [...prev]; next[zoneIdx] = newNodeId; return next })

This avoids the parent index confusion entirely because it works directly on the visual cardOrder.

### In ViewerInner
- Remove the `replaceIdx` logic from handleFocusFile's SplitGrid branch
- Add `onSpawnCard` callback that calls `buildNode` + `addNodes` and returns the node.id
- Pass `onSpawnCard` to SplitGrid
